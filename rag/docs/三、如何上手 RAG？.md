# 三、如何上手 RAG？

说实话，我见过太多人一上来就想搞"多模型协同""高级 Agent 编排"，结果连最基本的问答都跑不通。

RAG 这件事，最稳妥的路子是先做出一个能稳定回答问题、能看到效果、可评估的最小系统，然后再逐步优化召回、排序和生成质量。这不是我总结出来的经验，是我踩过坑之后才想明白的事。

我第一次搭 RAG 系统的时候，用了三天搭了一套"完美"的混合检索 + Rerank + 多路融合的架构，结果跑起来发现——连"公司年假多少天"这种最简单的问题都答不对。

排查了一圈，问题出在文档解析。原始 PDF 是扫描件，文字根本没提取出来，整套华丽的检索架构全是白搭。

这个故事告诉我一个道理：RAG 的上限，往往不是由模型决定的，是由数据质量决定的。

## 先把工具选对

工欲善其事，必先利其器。这部分不展开讲太多，只说我自己用下来觉得值得关注的几个选择。

**开发框架这块**，如果你们团队已经有 Spring Boot 项目，我建议直接上 Spring AI。跟现有体系集成自然，文档质量也不错，不用再学一套新东西。如果你想快速拼一条能跑通的 RAG 链路试试效果，LangChain4j 是更好的选择，组件比较全，社区也活跃，上手快。

至于完全原生开发——直接基于向量数据库 SDK 加 LLM API 自己拼——我建议等你在其他方案上踩过一轮坑之后再考虑。灵活性是好，但很多底层的东西你需要自己处理，初期没必要给自己加难度。

**向量数据库这块**，我自己的选择逻辑是这样的：

本地快速验证阶段，用 FAISS 就够了，轻量，不用搭服务，直接跑。但 FAISS 不适合做成分布式，生产环境跑一段时间肯定要换。

生产环境里，Milvus 是最常见的选择，性能强，分布式支持好，社区成熟，遇到问题好查资料。如果你团队本来就有 Elasticsearch 体系，而且需要混合检索（关键词加向量一起搜），那直接用 ES 的向量检索能力也行，省一套系统的运维成本。pgvector 适合那种团队 PostgreSQL 用得很熟、业务规模又不是特别大的场景，维护成本低，一个数据库搞定所有事。

**Embedding 模型这块**，说真的，选什么直接影响你检索效果的天花板，值得认真对待。

OpenAI 的 text-embedding-3 效果稳定，使用简单，但如果你的数据敏感不能上云，这条路就走不通了。这种情况下可以考虑 Hugging Face 上的开源模型，比如 bge-large-zh 或者 m3e-base，可以私有化部署，数据不外流。国内的话智谱 AI 和通义千问的 Embedding API 也都可以，便捷性上有优势。

我的建议是先用通用模型跑通全链路，确认效果有提升空间之后，再去测试专用模型。一步到位往往意味着你同时踩多个坑，哪个因素在影响效果都分不清。

## 四步搭出一个能用的 RAG 系统

### 第一步：把数据准备好、清洗干净

很多人一上来就"接模型"，数据还没整明白就开始调参，这是本末倒置。

数据质量直接决定检索质量的上限。你喂进去的是 garbage，出来的答案不可能不是 garbage。这话听着像废话，但真正做的时候特别容易忽略。

**文档解析是第一个坑。**

PDF、Word、Excel、HTML，不同格式的文档有不同的解析方式。最烦人的是扫描版 PDF——你以为文字在文档里，其实只是一张图片，普通的文本提取根本读不到任何内容。这种情况需要 OCR 处理，否则你辛苦建的索引里全是空白。

我的建议是先用几份有代表性的真实文档跑一遍解析流程，看看能提取出多少内容，提取出来的内容格式对不对，再往下走。

**分块策略是第二个坑。**

文档不能一股脑塞进去，太长了。一般会用文本分割器把文档切成小片段，每个片段单独算向量存进去。切成多大、要不要有重叠，这个没有万能公式，但有几个经验可以参考。

按语义段落切比按固定字符数切效果好。相邻块保留一定比例的重叠区间，减少上下文断裂。每块控制在 500 到 1000 tokens 左右比较常见，但最终还是得看你的实际场景——法律文档和聊天记录的分块策略肯定不一样。

还有一个很多人忽略的点：**给每个块保留元数据**。来源文档、页码、标题、版本号、更新时间，这些信息在后面溯源和过滤的时候非常有用。比如你不想让 AI 引用过期的制度文档，在检索层就可以根据版本号做过滤。

### 第二步：把索引建起来

数据准备好了，下一步是把向量索引建起来。

这一步的流程是：把切分好的文本片段通过 Embedding 模型转换成向量，然后连同原始文本和元数据一起存进向量数据库。

**增量索引一定要做。**

如果你只支持全量重建，那文档每次更新都要重新跑整个流程，时间成本很高。增量索引的意思是，只对新增或修改的文档做处理，已经索引过的内容不用动。这个机制越早做越好，等你系统跑起来了再加会很麻烦。

**高价值文档优先保证索引质量。**

刚起步的时候不要追求全量接入，选几份最高频使用的文档先把链路跑通，确保效果，再慢慢扩大范围。全量接入听起来很美好，但问题也会成倍增加，排查起来很痛苦。

### 第三步：把检索策略调对

这一步直接决定用户问题能不能被正确回答。

**不要只依赖单一向量检索。**

我见过有人从头到尾只用向量相似度搜索，然后抱怨说"这个专有名词怎么都搜不到"。向量检索擅长的是语义相似性，但用户的表达方式和文档里的说法往往不一样——你问"年假怎么算"，文档里写的是"带薪假期管理制度"，纯粹靠向量匹配有时候就是匹配不上。

我的建议是**向量检索加关键词检索混合使用**。两路并行，各有侧重，最后把两路的结果合并起来去重排序。

**Rerank 是值得加的一步。**

向量检索是第一轮海选，找出 Top-K 个候选，然后 Rerank 模型把这些候选按跟问题的真实相关性重新排序，把最相关的往前挪。这一步成本不高，但对最终效果提升很明显。

**查询改写也值得做。**

用户的口语化问法、缩写、内部术语，这些直接拿去做检索往往效果不好。加一层查询改写，把问法转换成更精准的检索词，或者把复杂问题拆成几个子问题分别检索。效果立竿见影。

**权限过滤别忘了。**

召回来了但用户没有权限看，这等于没召回。在检索层就把权限过滤做掉，不要把这个问题留到生成阶段。

### 第四步：把生成和 Prompt 调好

链路通了，下一步是让模型回答得更准、更有用。

这部分我踩过最大的坑是——**塞了太多上下文给模型**。

"context window 都 100 万 token 了，疯狂往里塞"，我之前也干过这事。结果模型反而被干扰了，答非所问。喂东西这件事，从来不是越多越好。

控制注入上下文的数量，把真正相关的内容优先塞进去，相关性低的能删就删。

还有一点很重要：**约束模型"找不到证据就直接说不知道"**。

这是减少幻觉最直接的方式之一。很多模型在没有足够信息的时候会选择"编一个看起来合理的答案"，你得明确告诉它：不要编，找不到就说找不到，你诚实承认不会答比瞎编强一百倍。

要求答案附带引用来源也是个好习惯。模型说"根据 XX 文档"，用户可以点进去核实，这本身也是一种幻觉的兜底。

![RAG MVP 流程](imgs/rag_mvp.svg)

这套 MVP 跑起来之后，怎么判断它是否达标？

能基于知识库正确回答明确的问题，比如"公司年假多少天"；能返回引用来源或者文档片段，用户能核实；面对证据不足的问题，能拒答或者明确说明依据不足；最后，你得有一套测试问题集，每次改动之后跑一遍，确保效果变化是可观察的。

如果这四条都能满足，你的基础系统就算跑通了。

## 两条上手路径，看你是什么角色

如果你是业务方的同学，想快速验证一下 RAG 这件事对你有没有价值，推荐走低代码路径。Dify 或者 FastGPT 都挺适合的，可视化操作，上传文档、配问答、跑效果，一个下午就能看到结果。MaxKB 是国产的，部署快，适合内部做 PoC。

如果你是技术团队，想把 RAG 接入现有的 Java 系统里，需要跟权限、审计、监控这些治理模块一起跑，那我建议走工程化开发路径。LangChain4j 或者 Spring AI 都可以，前者上手快，后者跟现有体系集成更自然。

这两条路没有高下之分，只有适用不适用。先验证业务价值比什么都重要——如果业务价值验证不出来，工程化做得再漂亮也是白搭。

## 落地过程中最容易踩的坑

搭完 MVP 之后往生产走，你会发现一堆在 MVP 阶段根本想不到的问题。我列几个最常见的，你提前知道能少走很多弯路。

**文档解析的坑。**

PDF 格式太乱了，扫描件、图片类内容提取不完整，流程图架构图只提文字会漏掉大量信息。我的建议是对高价值文档做 OCR 处理和人工抽检，重要图表提取描述和结构化信息。解析质量这件事，值得认真对待，不是随便调个库就完事了。

**分块策略的坑。**

块太大召回粗糙，块太小语义破碎。我自己踩出来的经验是，按章节和段落切比按固定字符数切效果好，但最终还是要根据你的文档类型反复调。固定 top_K 也容易出问题——有的问题需要召回更多内容才能答好，有的问题召回太多反而引入噪声。我的做法是用相关性阈值召回，而不是固定数字。

**专有名词和内部术语的坑。**

缩写、内部产品名、行业黑话——这些词向量检索非常不友好。一个我试过有效的方法是维护一份术语词典，把缩写映射到全称，把内部黑话映射到标准说法，检索之前先过一遍这个词典。效果提升很明显。

**新旧版本并存的坑。**

技术报告更新了，旧版本还没下线，结果 AI 引用了过期的内容，用户被误导了。这个问题在高变化性的文档场景里特别常见。我的做法是给每个文档加版本号和生效时间字段，排序的时候优先最新有效版本，答案里明确标注版本来源。

**短问题和精确关键词的坑。**

用户问"什么是 RAG"，四个字，向量检索对这种短文本效果很差，因为语义信息太少。这种情况加一路关键词检索进去，或者干脆给短问题单独处理，会有不错的提升。

## 怎么评估你的 RAG 系统？

搭完了，得知道它到底好不好，好在哪里、差在哪里。评估这件事很多人不重视，但它是系统持续优化的基础。

有几个指标我建议重点关注。

**文档召回率**是最核心的——对于一个问题，所有真正相关的文档里，有多少被系统成功检索出来了。这个指标如果太低，后面生成再强也是在错误的基础上认真回答。

**上下文精确度**看的是检索结果里真正相关的内容有没有排在前面。你召回了 10 个片段，但最相关的那个排在第 8 位，模型可能因为上下文长度限制根本没看到它。

**忠实度**说的是模型生成的答案有没有忠于它看到的上下文，有没有自己发挥、引入上下文里没有的信息。这个指标直接反映幻觉问题的严重程度。

**答案相关性**是最终答案跟用户原始问题的匹配程度。上下文召回了、答案也忠实了，但最后答非所问，那前面的努力全白费。

我的做法是准备一批高质量的问答集作为离线评测集，固定周期跑一遍看指标变化。每次改动分块策略、检索参数或者 Prompt 之前，先跑一遍基准测试，确保改动是有效的，而不是优化了一个点伤到了另一个点。

线上的话，结合人工反馈、点击率、采纳率、追问率这些业务指标一起看。技术指标和业务指标有时候会背离，最终还是要为业务效果负责。

## 写在最后

RAG 不是一个"接上向量库就结束"的事。数据质量、检索策略、评估闭环，这三件事做好了，系统才能真正用起来、持续优化下去。

我的建议是：**不要在一开始就追求完美**。先跑通全链路，看到效果，建立起评估体系，然后一个点一个点优化。这个顺序比什么都重要。
