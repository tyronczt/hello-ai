# 六、文本分块

## 1. 什么是RAG中的文本分块（Text Chunking）？

文本分块（Text Chunking）是构建 RAG 流程的关键步骤。它的原理是将加载后的长篇文档，切分成更小、更易于处理的单元。这些被切分出的文本块，是后续向量检索和模型处理的基本单位。

## 2. 为什么要进行文本分块？

### 2.1、突破大语言模型的长度限制

大型语言模型（LLM）的输入有“上下文窗口”限制，不能直接处理整本百科全书。若不进行分块，超长文档会被强制截断，导致信息丢失。

### 2.2、实现精准“大海捞针”，提升检索效率与精度

RAG 的核心是“先检索，后生成”。如果让 AI 在一整篇 10 万字的文档中寻找几句关键信息，不仅计算量巨大、响应缓慢，还会因为信息被稀释导致答案不准确。
相反，将文档预先切分成语义相关的小块，AI 就能在向量空间中快速、精准地定位到最相关的信息。

### 2.3、保留完整语义，避免上下文断裂

糟糕的分块如同用尺子把“我喜欢吃苹果”机械地切为“我喜欢吃苹”和“果”，会使 AI 无法理解其含义。好的分块策略会遵循语义边界（如句子、段落、主题），确保每个块内部信息完整。
为防止跨块信息丢失，常引入 overlap（重叠） 机制，让相邻块共享部分内容以保证上下文连贯，重叠比例通常为 10%-20%。高质量的文本块是 LLM 进行准确推理和生成答案的优质“燃料”。

### 2.4、支撑高效索引，奠定向量检索基础

RAG 系统的“记忆力”来自向量数据库。分块是将文本转为向量并存入数据库的前提。只有当文档被切分成信息密度高、语义清晰的文本块，并生成高质量的向量时，AI 才能基于语义相似度“回忆”起正确的知识。

### 2.5、实现检索粒度的灵活匹配

分块让你可以像调整搜索引擎的“精度旋钮”，根据场景灵活调整策略：

- 小文本块（约 256-512 tokens）：信息密度极高，适合需要精确匹配的事实类查询（如“产品A的保修期是多久？”），但上下文较少。
- 大文本块（约 512-1024 tokens）：包含更丰富的背景信息，适合需要宏观理解的上下文繁重型任务（如“总结这份报告的要点”），但可能引入噪声。

### 2.6、挑战与权衡

在选择分块策略时，也需考虑其在计算成本（如语义分块）、实现复杂度（如智能体分块）、通用性以及语义边界确定等方面的不同侧重。

## 3. 几个关键参数：chunkSize、overlap

在动手切之前，有两个参数必须搞清楚：`chunkSize`（块大小）和 `overlap`（重叠量）。

#### 3.1 chunkSize 怎么理解

chunkSize 就是每个块的长度上限。比如你设 chunkSize = 200，意思是每个块最多包含 200 个字符（或 200 个 token，取决于你用什么单位，后面会说）。

chunkSize 设多大合适？这没有标准答案，但有一个基本的权衡：

* 块太大（比如 2000 字）：每个块包含的信息多，但检索时容易混入不相关的内容，精度下降。就像用户问退货政策，结果返回了一整章包含退货、换货、维修的内容，模型还得自己从里面挑。
* 块太小（比如 50 字）：每个块很精准，但可能把一个完整的意思切断了，上下文丢失。就像把一条退货规则从中间劈开，前半句和后半句单独看都不知道在说什么。

一般来说，200 到 1000 个字符是比较常见的范围，具体取决于你的文档类型和检索需求。

#### 3.2 overlap 是什么，为什么需要它

overlap（重叠）是指相邻两个块之间共享的文本长度。

打个比方：你在看一本小说，每次只能记住一页的内容。如果你严格按页翻，第 1 页看完翻到第 2 页，那第 1 页最后一句话和第 2 页第一句话之间的联系就断了。但如果你每次翻页时，把上一页最后几行重新看一遍，这几行就是重叠的部分，它帮你保持了上下文的连贯性。

用一个具体的例子来看。假设知识库里有这样一段退货政策：

> 自签收之日起 7 天内，商品未经使用且不影响二次销售的，消费者可申请七天无理由退货。生鲜食品、定制商品、贴身衣物等特殊品类不适用此规则，具体以商品详情页标注为准。

如果 chunkSize = 40，不加 overlap，切出来可能是：

* 块 1：`自签收之日起 7 天内，商品未经使用且不影响二次销售的，消费者可申请七天`
* 块 2：`无理由退货。生鲜食品、定制商品、贴身衣物等特殊品类不适用此规则，具体`
* 块 3：`以商品详情页标注为准。`

注意看，“七天无理由退货”这个关键词被切成了两半，分别落在块 1 和块 2 里。用户搜“七天无理由退货”，两个块都匹配不完整。

##### 3.2.1 不加 overlap 会丢失什么

上面的例子已经很直观了——不加 overlap，相邻块的边界处一定会丢失上下文。如果用户恰好问的问题涉及到边界处的内容，检索就可能找不到完整的答案。

加上 overlap = 15 之后，切出来变成：

* 块 1：`自签收之日起 7 天内，商品未经使用且不影响二次销售的，消费者可申请七天`
* 块 2：`消费者可申请七天无理由退货。生鲜食品、定制商品、贴身衣物等特殊品类不适`
* 块 3：`特殊品类不适用此规则，具体以商品详情页标注为准。`

块 2 的开头和块 1 的结尾有重叠，“七天无理由退货”在块 2 里是完整的了。

当然，overlap 也不是越大越好。overlap 太大意味着大量重复文本，存储和计算成本都会上升。一般经验是 overlap 设为 chunkSize 的 10%\~25%。

#### 3.3 用什么单位：字符 vs token

你可能注意到了，前面说 chunkSize 的时候，有时说字符，有时说 token。这两个东西不一样，有必要区分一下。

字符（Character）就是你肉眼看到的每一个符号。“你好”是 2 个字符，"Hello"是 5 个字符，空格和标点也各算 1 个字符。

Token 是大模型实际处理文本的最小单位。大模型不是一个字一个字地读文本的，它会先把文本切成一个个 token。对于英文，一个常见单词通常是 1 个 token，长一点的单词可能被拆成 2-3 个 token。对于中文，一个汉字通常是 1-2 个 token。

举个例子：


| 文本           | 字符数 | 大约 token 数 |
| -------------- | ------ | ------------- |
| 七天无理由退货 | 7      | 7-10          |
| Hello World    | 11     | 2             |
| 退货政策       | 4      | 4-6           |

为什么要关心这个？因为大模型的上下文窗口是按 token 算的，不是按字符算的。如果你用字符数来设 chunkSize，实际消耗的 token 数可能比你预期的多（尤其是中文场景）。

不过在入门阶段，用字符数来设 chunkSize 完全够用。等你对 token 有了更深的理解，再考虑切换到基于 token 的分块也不迟。

## 4. 如何进行文本分块？

根据文档类型和任务选择合适策略是成功关键，以下是几种主流策略：


| 策略类型                                     | 核心原理                                                     | 优点                                             | 缺点                                                         |
| -------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------ | ------------------------------------------------------------ |
| **固定大小分块-Fixed Size Chunking**         | 按预设的字符数或 Token 数机械切分。                          | 简单、速度快、可预测。                           | 易破坏语义完整性。                                           |
| **重叠分块-Overlapping Chunking**            | 相邻块之间保留一定比例（如 10%-25%）的重复内容，以保持上下文连贯。 | 有效减少边界信息丢失，提升检索召回率。           | 存储成本增加，块之间存在冗余内容。                           |
| **递归分块-Recursive Chunking**              | 按优先级顺序（段落→句子→空格→字符）递归切分，直至块大小符合要求。 | 在满足大小限制的同时，最大限度保留文档逻辑结构。 | 实现稍复杂，有一定性能开销。                                 |
| **语义分块-Semantic Chunking**               | 计算句子 Embedding 的相似度，在主题切换处（相似度下降）切分。 | 语义高度连贯，检索质量高。                       | 计算成本高，速度慢。                                         |
| **滑动窗口分块-Sliding Window Chunking**     | 通过滑动窗口创建重叠块，窗口每次滑动固定步长，确保相邻块共享内容。 | 保持跨部分上下文连贯，避免边界信息丢失，实现简单。 | 块间冗余度高，存储成本增加，可能产生过多相似块影响检索精度。 |
| **模式特定分块-Content-Specific Chunking**   | 按内容类型（如文本、表格、代码、图像）分别处理，每个类型按其性质分块。 | 针对不同内容类型优化，保留特定格式的结构和语义。 | 需要预先识别内容类型，实现复杂，通用性较弱。                 |
| **上下文增强分块-Context-Enriched Chunking** | 为每个块添加周围块的摘要或元数据（如标题、章节信息），保持序列间的上下文。 | 增强块的上下文信息，提升检索和理解质量，适合长文档。 | 元数据生成增加计算成本，块大小难以控制，可能超出模型限制。   |
| **子文档分块-Sub-document Chunking**         | 先总结整个文档或大章节，将摘要作为元数据附加到各个子块，适合大型文档集。 | 保留宏观语义结构，块间关联性强，便于文档级检索。 | 需要额外的摘要生成步骤，实现复杂，存储和计算成本较高。       |
| **基于文档结构分块-Document Based Chunking** | 利用标题、表格、页码等文档自身结构进行切分。                 | 贴合文档逻辑结构，信息组织性强。                 | 依赖高质量文档解析，通用性弱。                               |
| **智能体 (Agentic) 分块-LLM-based Chunking** | 由 Agent 根据任务（如总结、问答）动态决定如何切分。          | 灵活性和针对性强。                               | 实现复杂，目前不普及。                                       |
| **混合分块-Hybrid Chunking**                 | 结合多种分块策略，如 Token 分块 + 语义相似度合并。           | 综合多种策略优势，兼顾效率与语义完整性。         | 实现复杂度较高，参数调优复杂。                               |

### 5.1、固定大小分块（Fixed Size Chunking）

固定大小分块是一种简单的分块策略，它按照预设的字符数或 Token 数将文档机械切分。
它不管你这一刀切在段落中间还是句子中间，到了字数就切。这既是它的优点（简单），也是它的缺点（可能切断语义）。


| 维度   | 说明                                                                               |
| ------ | ---------------------------------------------------------------------------------- |
| 优点   | 实现极其简单，性能好，不需要任何 NLP 处理                                          |
| 缺点   | 完全忽略文本结构，容易把句子、段落从中间切断，导致语义不完整                       |
| 适合   | 文本结构不重要的场景，比如日志文件、纯数据文本；或者作为其他策略的兜底方案         |
| 不适合 | 有明确段落结构的文档（知识库、产品手册、政策文件），因为切断语义会严重影响检索质量 |

### 5.2、重叠分块（Overlapping Chunking）

重叠分块是在固定大小分块的基础上，让相邻的块之间共享一部分内容。这种策略的核心目的是解决边界信息丢失的问题。

假设你设置 `chunkSize = 500`，`overlap = 100`，那么：

- 块 1：包含文档的第 1-500 个字符
- 块 2：包含文档的第 401-900 个字符（从第 401 个字符开始，与块 1 重叠 100 个字符）
- 块 3：包含文档的第 801-1300 个字符（与块 2 重叠 100 个字符）
- 以此类推...

**重叠分块的工作原理：**

```
文档：[ABCDEFGHIJKLMNOPQRSTUVWXYZ...]

固定大小分块（chunkSize=10）：
块1: [ABCDEFGHIJ]
块2: [KLMNOPQRST]
块3: [UVWXYZ...]

重叠分块（chunkSize=10, overlap=3）：
块1: [ABCDEFGHIJ]
块2: [HIJKLMNOPQR]  ← 与块1重叠 "HIJ"
块3: [PQRSTUVWXYZ]  ← 与块2重叠 "PQR"
```

**为什么需要重叠？**

考虑这样一个场景：有一段文本描述"北京故宫是中国明清两代的皇家宫殿，位于北京市中心"。如果恰好在"北京"和"故宫"之间切开：

- 块 1 结尾："...位于北京"
- 块 2 开头："故宫是中国..."

用户搜索"北京故宫"时，两个块都无法完整匹配。有了重叠，关键信息就能在至少一个块中保持完整。

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 有效减少边界信息丢失，提升检索召回率；实现简单，只需在固定分块基础上添加偏移量 |
| 缺点   | 存储成本增加（约增加 overlap/chunkSize 的比例）；块间存在冗余内容，可能引入噪声 |
| 适合   | 需要高召回率的场景，如法律条文检索、医疗诊断知识库；连续性强、上下文依赖重的文本 |
| 不适合 | 存储资源极度受限的场景；文本块独立性要求高、重复内容会影响检索精度的场景 |
| 参数建议 | overlap 通常设为 chunkSize 的 10%-25%，过小的 overlap 效果不明显，过大则成本激增 |

### 5.3、【常用】递归分块（Recursive Chunking）

递归分块是目前实践中最常用的策略。它的思路可以用一句话概括：先尝试用最大的分隔符切，切完如果某个块还是太大，就换一个更小的分隔符继续切，直到所有块都在 chunkSize 以内。

具体来说，它维护一个分隔符列表，按优先级从高到低排列，过程如下：
![recursive-chunking.png](imgs/chunking/recursive-chunking.png)
这个先粗后细的过程就是递归的含义——不是一刀切到底，而是逐层细化。

为什么这种方式好？因为它尽最大努力保留文本的结构。能按段落切就按段落切，段落太长了才按句子切，句子还太长才按逗号切……只有在万不得已的时候才会像固定大小分块那样按字符硬切。


| 维度   | 说明                                                                                |
| ------ | ----------------------------------------------------------------------------------- |
| 优点   | 兼顾了语义完整性和块大小控制，是目前最通用的分块策略                                |
| 缺点   | 分隔符列表需要根据语言调整（中文和英文的标点不同）；依赖文本中存在合理的分隔符      |
| 适合   | 绝大多数场景，尤其是你不确定该用什么策略的时候，递归分块是最安全的默认选择          |
| 不适合 | 对分块有特殊要求的场景，比如代码文件（需要按函数/类来切）、表格数据（需要按行来切） |

### 5.4、语义分块（Semantic Chunking）

前面策略有一个共同的局限：它们都是基于规则的——要么按字数切，要么按标点符号切。它们不理解文本在说什么。

语义分块换了一个完全不同的思路：用 Embedding 模型来判断文本的语义相似度，在语义发生明显变化的地方切割。

具体过程是这样的：

1. 先把文本按句子拆开（这一步可以简单地按句号切）
2. 对每个句子生成一个向量（Embedding）
3. 计算相邻句子之间的向量相似度
4. 当相邻句子的相似度低于某个阈值时，说明话题发生了转换，在这里切一刀

![semantic-chunking.png](imgs/chunking/semantic-chunking.png)

这种方式的好处是显而易见的：它切出来的每个块在语义上是高度内聚的，不会出现一个块里混着两个不相关话题的情况。

相比 Embedding 计算相似度的方式，直接让大模型来做分块决策更加直观——大模型本身就具备强大的语义理解能力，可以直接读懂文本，判断哪里应该切分。

最直接的思路：把文本交给大模型，让它找出主题切换的位置。


| 维度   | 说明                                                                                               |
| ------ | -------------------------------------------------------------------------------------------------- |
| 优点   | 切割点基于语义而非规则，分块质量最高，每个块的主题高度内聚                                         |
| 缺点   | 需要调用 Embedding 或者 Chat 模型，有额外的计算成本和延迟；阈值需要调参；对模型的质量有依赖        |
| 适合   | 对检索精度要求很高的场景，比如法律文档问答、医疗知识库、金融合规文档                               |
| 不适合 | 文档量特别大且对延迟敏感的场景；文本本身结构已经很清晰的场景（用递归分块就够了，没必要上语义分块） |

### 5.5、滑动窗口分块（Sliding Window Chunking）

滑动窗口分块是一种特殊的重叠分块策略，它通过固定步长的滑动窗口来创建文本块。与普通的重叠分块不同，滑动窗口更强调"平滑过渡"和连续性。

**工作原理：**

滑动窗口分块有三个核心参数：
- `window_size`：窗口大小（每个块的长度）
- `stride`：步长（窗口每次滑动的距离）
- `overlap = window_size - stride`：重叠长度

```
文档：[ABCDEFGHIJKLMNOPQRSTUVWXYZ...]

滑动窗口分块（window_size=8, stride=3）：
块1: [ABCDEFGH]
块2: [DEFGHIJK]  ← 向右滑动3个字符
块3: [GHIJKLMN]  ← 再向右滑动3个字符
块4: [JKLMNOPQ]
...
```

**滑动窗口 vs 普通重叠分块：**

| 对比维度 | 普通重叠分块 | 滑动窗口分块 |
| -------- | ------------ | ------------ |
| 切分方式 | 按固定大小切分，相邻块共享尾部/头部 | 窗口平滑滑动，每个新块都是原窗口的平移 |
| 重叠控制 | 仅首尾重叠 | 整个窗口范围内都有不同程度的重叠 |
| 块数量 | 相对较少 | 相对较多（取决于 stride 大小） |
| 适用场景 | 一般文档检索 | 需要细粒度上下文保持的序列数据 |

**实际应用场景：**

1. **时间序列文本**：如聊天记录、日志流，需要保持时间上的连续性
2. **代码分析**：分析代码上下文时，需要看到前后关联的代码段
3. **语音识别转文本**：语音转写的文本需要保持说话内容的连贯性

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 保持跨部分上下文连贯，避免边界信息丢失；实现简单直观；适合序列型数据 |
| 缺点   | 块间冗余度高，存储成本显著增加；可能产生过多相似块，影响检索精度和效率；stride 参数需要仔细调优 |
| 适合   | 时间序列文本（聊天记录、日志）；需要细粒度上下文分析的代码；流式数据处理 |
| 不适合 | 存储资源受限的场景；需要高检索精度的场景（大量相似块会干扰排序）；独立性强的文档集合 |
| 参数建议 | stride 通常设为 window_size 的 50%-75%，较小的 stride 产生更多块但上下文更完整 |

### 5.6、模式特定分块（Content-Specific Chunking）

模式特定分块（也称为内容感知分块）的核心思想是：不同类型的内容应该采用不同的分块策略。文本、表格、代码、图像各有其最佳的分块方式。

**为什么需要模式特定分块？**

考虑一个包含多种内容的 PDF 文档：
- 正文段落：适合按段落或句子切分
- 数据表格：应该按行或按表格整体切分，不能把一行数据切开
- 代码块：应该按函数、类或逻辑块切分
- 图片及描述：图片和它的 caption 应该作为一个整体

如果用统一的分块策略处理，会破坏这些内容的内在结构。

**不同内容类型的分块策略：**

| 内容类型 | 分块策略 | 示例 |
| -------- | -------- | ---- |
| 普通文本 | 递归分块或语义分块 | 按段落→句子层级切分 |
| Markdown | 按标题层级切分 | #、##、### 作为分隔符 |
| HTML | 按标签结构切分 | `<div>`、`<section>`、`<p>` 等 |
| 表格 | 按行或按表切分 | 每行作为独立块，或整个表格+描述 |
| 代码 | 按函数/类/逻辑块切分 | AST 解析后按语法结构切分 |
| JSON/XML | 按节点切分 | 每个对象或数组元素作为块 |
| LaTeX | 按章节和环境切分 | `\section{}`、`\begin{}...\end{}` |

**代码分块的特殊处理：**

代码文件使用普通文本分块会产生严重问题——可能把一个函数从中间切断。正确的做法是：

```python
# 原始代码
def calculate_total(price, quantity):
    """计算总价"""
    discount = 0.9 if quantity > 10 else 1.0
    return price * quantity * discount

def apply_tax(amount, tax_rate):
    """应用税率"""
    return amount * (1 + tax_rate)
```

使用 AST（抽象语法树）解析后，可以按函数完整切分：
- 块 1：完整的 `calculate_total` 函数
- 块 2：完整的 `apply_tax` 函数

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 针对不同内容类型优化，保留特定格式的结构和语义；分块质量最高，专业文档处理效果好 |
| 缺点   | 需要预先识别内容类型，增加预处理复杂度；实现复杂，需要多种解析器；通用性较弱，新格式需要额外开发 |
| 适合   | 结构化文档（PDF、Word、HTML）；混合内容文档；代码仓库检索；技术文档和 API 文档 |
| 不适合 | 纯文本场景（增加不必要的复杂度）；快速原型验证；内容类型未知的文档流 |
| 实现建议 | 使用专门的解析库（如 BeautifulSoup 解析 HTML、tree-sitter 解析代码、pandas 处理表格） |

### 5.7、上下文增强分块（Context-Enriched Chunking）

上下文增强分块是一种"元数据增强"策略。它不为每个块添加周围块的内容，而是添加周围块的**摘要或关键元数据**，从而在控制块大小的同时保持上下文关联。

**核心思想：**

普通分块：
```
块内容：[产品的保修期为一年，自购买之日起计算。]
```

上下文增强后：
```
[章节：售后服务政策]
[前文摘要：本文档说明产品的退换货和保修政策]
块内容：[产品的保修期为一年，自购买之日起计算。]
[后文摘要：以下情况不在保修范围内...]
```

**元数据类型：**

| 元数据类型 | 说明 | 示例 |
| ---------- | ---- | ---- |
| 文档级元数据 | 整个文档的信息 | 标题、作者、创建时间、文档类型 |
| 章节级元数据 | 当前章节的信息 | 章节标题、层级、章节摘要 |
| 相邻块摘要 | 前后块的简短描述 | 前文讲了什么、后文将讲什么 |
| 关键词标签 | 块的核心主题 | #保修政策 #售后服务 |
| 时间/位置信息 | 时序或位置上下文 | 第3页、2024年第一季度 |

**实现方式：**

1. **预处理方式**：在分块前提取文档结构，将元数据注入每个块
2. **后处理方式**：分块后，用 LLM 为每个块生成上下文摘要
3. **混合方式**：结构元数据（标题、章节）+ 生成式摘要（内容概述）

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 增强块的上下文信息，提升检索和理解质量；块间关联性强，便于追踪信息来源；适合长文档和复杂查询 |
| 缺点   | 元数据生成增加计算成本；增强后的块大小难以控制，可能超出模型限制；需要额外的存储空间保存元数据 |
| 适合   | 长文档检索（论文、报告、书籍）；需要溯源的场景；复杂多跳问答；法律、医疗等专业领域 |
| 不适合 | 短文档或片段文本；对延迟极度敏感的场景；存储成本敏感的场景 |
| 最佳实践 | 元数据应简洁明了，控制在块大小的 20%-30%；优先使用结构化元数据（标题、章节），必要时再使用生成式摘要 |

### 5.8、子文档分块（Sub-document Chunking）

子文档分块是一种"分层摘要"策略。它先对整个文档或大章节生成摘要，然后将这些摘要作为元数据附加到各个子块上。这样每个子块都能"感知"到它所在的更大上下文。

**分层结构示意：**

```
文档级摘要
├── 章节1摘要
│   ├── 块1.1 (内容 + 章节1摘要 + 文档级摘要)
│   ├── 块1.2 (内容 + 章节1摘要 + 文档级摘要)
│   └── 块1.3 (内容 + 章节1摘要 + 文档级摘要)
├── 章节2摘要
│   ├── 块2.1 (内容 + 章节2摘要 + 文档级摘要)
│   └── 块2.2 (内容 + 章节2摘要 + 文档级摘要)
└── 章节3摘要
    └── ...
```

**工作流程：**

1. **第一层**：对完整文档生成高层摘要（100-200字）
2. **第二层**：将文档划分为大章节，为每个章节生成章节摘要（50-100字）
3. **第三层**：对每个章节进行常规分块
4. **增强**：将文档摘要和对应章节摘要附加到每个子块

**示例：**

假设有一篇关于"人工智能发展历史"的长文章：

```
【文档摘要】本文回顾了人工智能从1950年代至今的发展历程，
涵盖符号主义、连接主义、深度学习三个主要阶段。

【章节1摘要】符号主义阶段（1950-1980）：基于逻辑推理和专家系统。
块1.1: 1956年达特茅斯会议标志着AI诞生...[附：章节1摘要 + 文档摘要]
块1.2: 专家系统在医疗诊断中的应用...[附：章节1摘要 + 文档摘要]

【章节2摘要】连接主义阶段（1980-2010）：神经网络和机器学习方法。
块2.1: 反向传播算法的提出...[附：章节2摘要 + 文档摘要]
...
```

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 保留宏观语义结构，块间关联性强；便于文档级检索和跨块推理；每个块都有全局视野，减少"只见树木不见森林" |
| 缺点   | 需要额外的摘要生成步骤（通常需要 LLM）；实现复杂，需要多层处理；存储和计算成本较高；摘要质量影响最终效果 |
| 适合   | 大型文档集合（论文库、法规库、知识库）；需要跨章节推理的复杂问答；文档摘要和导航场景 |
| 不适合 | 短文档；实时性要求高的场景；简单的事实检索（增加不必要的开销） |
| 优化建议 | 摘要应聚焦"这个章节/文档在讲什么"而非细节；可以考虑使用轻量级模型生成摘要以降低成本；对重要文档可以人工审核摘要质量 |

### 5.9、基于文档结构分块（Document Based Chunking）

基于文档结构分块利用文档自身的结构标记（如标题、章节、页码、表格边界等）进行切分。这是一种"尊重原文"的分块策略。

**常见的文档结构标记：**

| 文档类型 | 结构标记 | 分块方式 |
| -------- | -------- | -------- |
| Word/DOCX | 标题样式（Heading 1/2/3）、段落 | 按标题层级切分 |
| PDF | 书签、页眉页脚、文本块位置 | 按页或按书签切分 |
| Markdown | #、##、### 标题标记 | 按标题层级切分 |
| HTML | h1/h2/h3、section、article | 按标签结构切分 |
| LaTeX | \section、\subsection | 按命令层级切分 |

**Word 文档分块示例：**

```
第一章 概述
  1.1 背景介绍
    正文内容...
  1.2 研究目的
    正文内容...
第二章 方法论
  2.1 实验设计
    正文内容...
```

按 Heading 1（一级标题）切分：
- 块 1：第一章 概述（包含 1.1 和 1.2 的内容）
- 块 2：第二章 方法论（包含 2.1 及后续内容）

按 Heading 2（二级标题）切分：
- 块 1：1.1 背景介绍
- 块 2：1.2 研究目的
- 块 3：2.1 实验设计
- ...

**PDF 文档的特殊处理：**

PDF 是"页面描述格式"而非"结构标记格式"，提取结构需要特殊处理：

1. **基于书签**：使用 PDF 的书签（目录）进行切分
2. **基于布局**：分析文本块的位置、字体大小推断标题层级
3. **基于页码**：简单按页切分（适合扫描版 PDF）

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 贴合文档逻辑结构，信息组织性强；分块边界自然，语义完整；保留文档的层次关系，便于导航 |
| 缺点   | 依赖高质量文档解析，解析失败则效果大打折扣；不同文档格式需要不同的解析器；扫描版/图片版 PDF 难以提取结构；通用性弱，对新格式支持需要开发 |
| 适合   | 结构化文档（正式报告、论文、书籍）；有明确章节层次的文档；需要保留文档导航结构的场景 |
| 不适合 | 非结构化文本（纯文本、聊天记录）；扫描版/图片版文档；格式混乱或结构不清晰的文档 |
| 工具推荐 | Python: python-docx (Word)、PyPDF2/pdfplumber (PDF)、BeautifulSoup (HTML) |

### 5.10、智能体分块（LLM-based / Agentic Chunking）

智能体分块是一种由大语言模型（LLM）或 AI Agent 动态决定如何切分的策略。它不依赖固定的规则，而是让模型"理解"文本后做出分块决策。

**核心思想：**

传统分块：按字数/标点/规则切分，模型不理解内容
智能体分块：让模型阅读文本，判断"哪里该切"

**实现方式：**

1. **直接分块法**
   将文本交给 LLM，让它直接输出分块边界：
   ```
   提示：请分析以下文本，在主题转换的地方标记 [CUT]。
   文本：...
   
   输出：
   第一段内容...[CUT]
   第二段内容...[CUT]
   第三段内容...
   ```

2. **主题识别法**
   先让 LLM 识别文本中的主题边界，再据此切分：
   ```
   提示：请识别以下文本中的主题转换点，输出每个主题的起止位置。
   ```

3. **任务驱动法**
   根据具体任务（问答、摘要、翻译）决定分块策略：
   - 问答任务：确保每个块包含完整的答案单元
   - 摘要任务：按论点/论据切分
   - 翻译任务：按语义完整的句子组切分

**Agentic 分块的进阶形态：**

更复杂的 Agentic 分块可以：
- 先浏览整个文档，生成文档地图
- 根据用户查询意图动态调整分块粒度
- 对不同类型的内容自动选择不同的分块策略
- 评估分块质量并迭代优化

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 灵活性和针对性最强；能处理复杂的语义边界；可根据任务动态调整；分块质量理论上最高 |
| 缺点   | 实现复杂，目前不普及；计算成本高（需要调用 LLM）；延迟大，不适合实时场景；对模型能力有依赖，模型质量直接影响分块效果 |
| 适合   | 对分块质量要求极高的场景；复杂文档（多主题交织、非线性结构）；需要任务自适应分块的场景；研究和实验性项目 |
| 不适合 | 大规模文档处理（成本过高）；实时性要求高的场景；预算有限的场景；简单结构文档（大材小用） |
| 未来展望 | 随着模型速度提升和成本下降，Agentic 分块可能成为主流；可以结合其他策略作为"精修"步骤 |

### 5.11、混合分块（Hybrid Chunking）

混合分块是结合多种分块策略的"组合拳"。它旨在综合各种策略的优势，在效率、成本和质量之间取得平衡。

**常见的混合策略组合：**

| 组合策略 | 第一层处理 | 第二层处理 | 适用场景 |
| -------- | ---------- | ---------- | -------- |
| 结构 + 语义 | 先按文档结构切分大段 | 对每个大段进行语义分块 | 长文档精细化处理 |
| 递归 + 语义 | 先用递归分块初步切分 | 对边界处进行语义相似度检查 | 提升分块质量同时控制成本 |
| 固定 + 智能体 | 先用固定大小快速分块 | 用 LLM 检查并调整边界 | 快速+精确的平衡 |
| 模式特定 + 增强 | 按内容类型分块 | 为每个块添加上下文元数据 | 复杂文档高质量检索 |

**混合分块示例（递归 + 语义）：**

```
步骤1：递归分块
├── 初步切分得到块A、块B、块C...

步骤2：边界语义检查
├── 检查块A末尾句子与块B开头句子的相似度
├── 如果相似度 > 阈值：合并块A和块B
├── 如果相似度 < 阈值：保持切分

步骤3：输出最终块
```

**分层混合策略：**

对于复杂的大型知识库，可以采用分层混合：

1. **文档层**：按文档类型选择基础策略（模式特定）
2. **章节层**：按文档结构切分大章节（文档结构分块）
3. **段落层**：对每个章节进行递归分块（递归分块）
4. **精修层**：对关键段落进行语义分块优化（语义分块）
5. **增强层**：为重要块添加上下文元数据（上下文增强）

| 维度   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 优点   | 综合多种策略优势，兼顾效率与语义完整性；灵活可调，可根据场景定制；通常能在成本和质量间找到最佳平衡点 |
| 缺点   | 实现复杂度较高；参数调优复杂（多种策略的参数需要协调）；调试困难，问题定位复杂；可能过度工程化 |
| 适合   | 生产环境的大规模 RAG 系统；对质量有要求但成本也需要控制的场景；复杂多样的文档集合 |
| 不适合 | 快速原型验证；简单场景（单一策略足够）；资源极度受限的环境 |
| 设计建议 | 从简单策略开始，根据实际需求逐步增加复杂度；建立评估指标，用数据驱动策略选择；优先保证核心场景的效果，边缘场景可以妥协 |
