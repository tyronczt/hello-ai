# 八、向量数据库

上一篇我们把文本 chunk 变成了向量，也顺手跑通了一个最小检索闭环。

当时其实还留了一个很关键的问题。

向量生成之后，存哪？

如果你做的只是一个 demo，这问题很简单，直接拿一个 `List<float[]>` 存在内存里就行。查询的时候遍历整个列表，逐个算余弦相似度，最后取 Top-K。

数据少的时候，这么干完全没问题。

但你把场景稍微放大一点，就会发现事情不对劲了。

一个电商知识库，商品说明、退货规则、物流政策、活动文案、FAQ，分块之后轻轻松松几十万，甚至上百万个 chunk。每个 chunk 后面还挂着一个几百维、上千维，甚至 4096 维的向量。

这时候用户每问一个问题，你都把查询向量拿去和库里几十万个向量逐个比较一遍？

这显然不现实。

所以这一篇要解决的，其实就是两个问题。

第一，向量到底该存在哪里。

第二，海量向量里最相似的那几个，到底怎么高效找出来。

## 为什么普通数据库不适合做大规模向量检索

### 最直觉的方案，就是拿 MySQL 硬存

很多人第一次接触向量数据库时，脑子里的第一个想法都很朴素。

向量不就是一组浮点数吗，那我存 MySQL 不就完了。

这个思路不是不能跑。

你完全可以在表里加一个 `JSON` 或 `TEXT` 字段，把向量序列化后存进去。

```sql
CREATE TABLE chunk_vectors (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    chunk_text TEXT NOT NULL,
    vector JSON NOT NULL,
    doc_id VARCHAR(64),
    category VARCHAR(32),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

然后查询的时候，把这些向量都读出来，在应用层逐个算余弦相似度，再排序取 Top-K。

如果只是做 demo，这么干完全能跑通。

问题出在数据一多，这套方案会立刻露馅。

### 真正的瓶颈，不是存，而是搜

假设你的知识库里有 100 万个 chunk，每个 chunk 对应一个 4096 维向量。每次用户提问，系统至少要做四件事。

1. 先把用户问题转成查询向量
2. 把库里的向量读出来
3. 逐个计算相似度
4. 排序后取 Top-K

这里最重的就是第三步。

一次余弦相似度计算，本质上就是对 4096 个维度做乘法、加法，再配上归一化相关计算。100 万个向量，就是 100 万次这样的计算。

这还没算 I/O。

因为你不只是算，还要先把大批向量从存储层读进来。

这时候你会发现，所谓的性能问题，不只是 CPU 忙，磁盘和内存也一起被拖下水了。

如果只有一个用户查询，可能还勉强能忍。

但只要并发一上来，这套方案立刻就会开始抖。

所以普通数据库的问题，不是完全不能存向量，而是它没有为高维相似度检索这件事做专门优化。

它擅长的是精确匹配。

不是近邻搜索。

### 暴力搜索为什么在生产里扛不住

暴力搜索也叫 Brute Force Search。

名字就已经把气质说明白了。

不绕路，不走捷径，老老实实把每个候选都比一遍。

它的好处当然也很直接。

准。

只要你把整个库都扫完，结果就是精确的。

但问题也同样直接。

慢。

你可以简单对比一下。

| 指标 | 暴力搜索 | ANN 检索 |
| --- | --- | --- |
| 100 万向量查询耗时 | 秒级 | 毫秒级 |
| 召回率 | 100% | 通常 95% 到 99% |
| 是否需要专门索引 | 不需要 | 需要 |
| 适用数据量 | 小规模 | 百万到亿级 |

这里最关键的，不是 ANN 完全准确，而是它在速度和效果之间做了一个工程上非常划算的交换。

你牺牲一点点理论最优解，换来几百倍甚至几千倍的性能提升。

在 RAG 场景里，这个交换通常非常值。

因为你本来也不是只拿一个结果直接交卷，而是会取一批 Top-K chunk，再交给大模型做综合理解。少掉某个排在第 27 或第 43 的 chunk，往往并不会明显影响最终回答。

所以向量数据库存在的核心原因，不只是它能存向量。

而是它能在海量向量里，快速找到最像的那几个。

一句话概括，向量数据库 = 向量存储 + ANN 索引 + 高效检索。

## ANN 到底是怎么做到又快又准的

既然暴力搜索扛不住，那接下来的问题就是，ANN 到底是怎么提速的。

答案其实不神秘。

核心思路就一句话。

不要每次都把全库翻一遍。

你可以把暴力搜索想成在一个 100 万人的城市里找和你最像的人，只能挨个问。ANN 则更像先按区域、特征、邻居关系把范围缩小，再在小范围里做更精细的搜索。

它不保证每次都找到绝对全局最优，但大多数时候能在极短时间内找到足够接近最优的答案。

工程上，这已经够用了。

## IVF 是怎么工作的，为什么叫先分区再搜索

IVF 的全称是 Inverted File Index。

名字看着挺学术，但你可以把它理解成先分区，再进区里找。

它的基本思路是，先用聚类把整个向量空间切成多个簇。每个簇都有自己的中心点，向量会被分配到离自己最近的那个簇里。

等查询来了，系统不会去扫全库，而是先看查询向量离哪些簇中心最近，然后只在最相关的几个簇里继续搜索。

这个思路特别像图书馆找书。

你要找一本 Java 并发相关的书，不会先把整个馆从头翻到尾，而是先去计算机区，再去编程语言区，再去 Java 那排书架。

搜索范围一下就缩小了。

IVF 的优点也很明确，结构直观，内存压力相对小，数据量大时比较稳。

但它也有代价。

如果某个向量刚好落在簇边界附近，就有可能被漏掉。还有就是参数调优，比如 `nlist`、`nprobe` 这些，调不好时效果会明显波动。

所以 IVF 更像一种偏稳健、偏节省资源的方案。

## HNSW 为什么几乎成了默认答案

如果你这两年看过 Milvus、Qdrant、pgvector、Weaviate 这些资料，大概率会经常看到 HNSW。

这不是巧合。

它现在基本就是 ANN 领域最主流的答案之一。

HNSW 的全称是 Hierarchical Navigable Small World Graph。

你不用记全称。

记住它的画面就行。

一张多层图。

顶层节点少，但跨度大，负责快速跳到目标附近。

底层节点多，但连接更密，负责精细搜索。

这个结构很像你在陌生城市里找人。

你不会一上来就满街乱问，而是先通过少量高层关系快速缩小范围，再一层层往下找，最后落到具体的人身上。

HNSW 做的也是这个事。

查询开始时，从高层图快速接近目标区域，再逐层下降，最后在底层图里找到最相似的候选。

它为什么这么火？

因为它在速度、召回率和工程可用性之间，平衡得特别好。

一般来说，HNSW 的优点是检索快、召回率高、效果稳，而且支持动态插入。

缺点也很清楚。

吃内存。

因为它不只是存向量，还要维护图结构里的大量边。数据一大，内存开销会明显上来。

所以 HNSW 很强，但不是免费午餐。

你是在用内存和建索引成本，换更高的检索性能。
![HNSW分层搜索](https://oss.open8gu.com/image-20260219211612679.png)

## IVF 和 HNSW 到底该怎么选

如果你刚开始做项目，这里最容易掉进一个坑。

一上来就想找最先进的算法。

但工程里真正该问的，其实不是谁最先进，而是谁最适合现在这套约束。

你可以先粗暴记一版。

| 索引类型 | 核心思路 | 速度 | 召回率 | 内存占用 | 更适合什么场景 |
| --- | --- | --- | --- | --- | --- |
| FLAT | 全量暴力搜索 | 最慢 | 100% | 低 | 数据量很小，极致追求精确 |
| IVF_FLAT | 聚类分区后簇内精确搜索 | 快 | 较高 | 较低 | 数据量大，内存有限 |
| IVF_SQ8 / IVF_PQ | 聚类 + 压缩量化 | 更快 | 略低 | 更低 | 数据量特别大，要省内存 |
| HNSW | 多层图导航搜索 | 很快 | 很高 | 高 | 百万级检索，追求速度和效果平衡 |
| DISKANN | 磁盘友好的图索引 | 较快 | 较高 | 低 | 亿级数据，内存放不下 |

如果数据量很小，比如十万以内，其实 FLAT 直接扫也不是不行。

如果已经到百万级，而且内存还算充足，HNSW 往往是很稳的默认选项。

如果数据规模继续往上走，或者内存预算明显受限，IVF 系列、量化压缩方案、DISKANN 这类路线就会更有吸引力。

所以选索引，本质上是在平衡四件事。

**速度。**

**召回率。**

**内存。**

**数据规模。**

## 向量数据库和传统数据库，到底差在哪

很多人会把向量数据库理解成「能存向量的数据库」。

这句话不能算错，但还是太轻了。

真正的区别不在于能不能存，而在于它们面对的问题完全不同。

传统数据库擅长的是结构化数据和精确匹配。

比如 `WHERE age = 25`、`status = 'paid'`、`order_id = '2026012345'` 这种查询，它们很拿手。

但你要它在一个高维向量空间里做相似度搜索，那就等于让它去做本来没为之设计的事。

而向量数据库从设计开始，就是冲着近似最近邻检索去的。

所以你可以这样理解。

| 维度 | 向量数据库 | 传统数据库 |
| --- | --- | --- |
| 核心数据类型 | 高维向量 | 结构化字段 |
| 核心查询方式 | 相似度搜索 | 精确匹配 |
| 索引机制 | HNSW、IVF 等 ANN 索引 | B-Tree、Hash 等 |
| 典型场景 | RAG、推荐、图像检索 | ERP、CRM、交易系统 |
| 一致性重点 | 检索效率与扩展性 | 事务与强一致性 |

不过它们不是替代关系。

更像分工关系。

在一个真实 AI 系统里，结构化业务数据通常还是放传统数据库里，向量数据和近似检索能力交给向量数据库。两边配合，才是常见方案。

## 主流向量数据库怎么选，不要一上来就无脑上最重的

讲到这里，另一个问题就来了。

向量数据库这么多，到底选谁。

这件事没有标准答案。

更多是看你的团队现状、数据规模、运维能力和系统目标。

### pgvector，适合已经在用 PostgreSQL 的团队

pgvector 不是独立数据库，而是 PostgreSQL 的扩展。

这点特别关键。

因为它意味着，如果你的团队本来就在用 PostgreSQL，那你不需要为了做向量检索额外再引一套数据库系统。

直接扩展就能用。

它的优点很明显，架构简单，运维沿用原有 PostgreSQL 体系，SQL 和标量过滤结合得也很好。

但它的边界也很清楚。

适合中小规模。

如果你真的往超大规模走，它不是最能打的那个。

### Chroma，适合原型验证和教学场景

Chroma 很轻。

这既是优点，也是边界。

它特别适合本地开发、实验、课程 demo、小型产品验证。API 简单，上手快，几行代码就能跑起来。

但你真要拿它去扛高并发、超大规模生产场景，心里最好别抱太高预期。

### Milvus，适合中大规模生产环境

Milvus 基本可以算开源向量数据库里最成熟的一档。

它本来就是冲着大规模向量检索去设计的，支持多种索引类型、支持分布式、生态也比较完整。

如果你是 Java 技术栈，或者明确知道后面数据规模会上去，Milvus 往往是一个非常稳的选择。

代价就是部署和运维复杂度会高一些，对硬件资源要求也更高。

所以它不是那种「开箱就轻松」的路线，而是「值得认真投入」的路线。

### Qdrant，适合想兼顾性能和易用性的团队

Qdrant 这几年也很强势。

它的特点是性能不错，接口友好，部署比 Milvus 轻一些，元数据过滤能力也比较强。

如果你的数据规模不算极端夸张，但又希望系统别太笨重，Qdrant 会是个很有竞争力的选择。

### Elasticsearch，不是纯向量库，但混合检索很香

Elasticsearch 严格来说不是原生向量数据库起家的，但它支持 dense vector 之后，在企业里就变得很有吸引力。

原因很现实。

很多公司本来就有 ES 集群。

这时候你想补上语义检索能力，不一定非得再单独起一套向量数据库。直接基于现有 ES 做关键词 + 向量的混合检索，落地成本会低很多。

所以如果团队已经深度使用 ES，这条路线经常非常划算。

## 选型时到底该看哪些维度

如果你不想被一堆产品名绕晕，最简单的办法是抓几个真正关键的判断维度。

| 维度 | 你真正该看什么 |
| --- | --- |
| 部署复杂度 | 你们团队能不能搞定运维和集群 |
| 检索性能 | 延迟、吞吐、召回率是否达标 |
| 扩展性 | 数据规模上涨后能不能水平扩展 |
| 集成成本 | 能不能和现有技术栈自然结合 |
| 高级能力 | 元数据过滤、混合检索、多索引支持 |
| 社区生态 | 文档、SDK、活跃度、问题修复速度 |

这几个维度里，我觉得最容易被忽略的是集成成本。

很多时候，性能不是唯一指标。

如果你团队本来就有 PostgreSQL 或 Elasticsearch，直接在现有体系上扩展能力，往往比额外引入一套全新系统更划算。

所以别动不动就追最强。

先看你现在手上有什么牌。

## 小结

这一篇其实主要解决了一个问题。

为什么 RAG 里不能只把向量随便找个地方存起来，而是需要专门的向量数据库。

原因很简单。

真正难的从来不是存，而是搜。

当数据量和并发一上来，暴力搜索很快就会失去实用价值。向量数据库真正提供的核心能力，是 ANN 索引和高效检索。

你如果只记住几件事，我觉得够了。

第一，普通数据库能存向量，但不适合做大规模高维近邻检索。

第二，ANN 的核心价值，是用很小的精度代价换极大的性能收益。

第三，HNSW 往往是百万级 RAG 场景下很稳的默认选项，但也要接受它更吃内存。

第四，数据库选型没有标准答案，核心还是看数据规模、团队能力和现有技术栈。

到这里，RAG 的离线链路又补上了一块。

文档提取、分块、向量化、向量存储，这几步已经能串起来了。

但还有一个问题没解决。

如果用户的问题里带着订单号、时间、商品编号这种精确信息，只靠纯向量检索其实不够。

下一篇，我们就该聊检索策略了，比如 BM25、混合检索、Rerank，这些东西怎么一起把召回质量往上拉。
