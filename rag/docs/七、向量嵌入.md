# 七、向量嵌入

最近有个朋友问我，说他看 RAG 相关的文章，每次看到"向量嵌入""Embedding""余弦相似度"这些词就懵。问我能不能用人话解释一下。

我说行啊，但解释之前得先问清楚——你是只想知道向量嵌入是什么，还是想知道向量嵌入在 RAG 里怎么用，以及怎么选模型？

他说都要。

那这篇文章，我就假设屏幕前的你跟他差不多，不讲那些学术定义，直接用大白话搞定。

## 向量嵌入是啥

说白了，Embedding 就是输入一段文本，输出一组浮点数向量。

这组数有一个固定的长度，比如 1024 个数字。这 1024 个数字，就是这段文本在"语义空间"里的坐标。

你想想，地图上的每个位置用经纬度来标记，Embedding 做的事差不多——它给每个文本一个坐标。只不过这个坐标不是二维的经纬度，而是几百甚至几千维的向量。

![embedding-progress.webp](imgs/embedding/embedding-progress.webp)

关键的地方来了。

语义上越像的文本，它们的坐标就越近。余弦相似度越高。

"你好"和"hello"意思差不多，向量距离就很近。

"苹果手机"和"iPhone"说的是一个东西，距离也近。

但"苹果手机"和"水果苹果"，距离就远。

这就是向量嵌入的核心能力——它把人类能理解的语义，转换成了计算机能计算距离的数字。

## 向量空间里的度量方式

刚说了语义相近的文本距离近，那具体怎么算这个"距离"？

主要有三种。

1. **余弦相似度** (Cosine Similarity)，计算两个向量夹角的余弦值。夹角越小，余弦值越接近 1，语义越像。这是向量检索里最常用的指标。
2. **点积** (Dot Product)，把两个向量对应位置上的数字乘起来，再加起来。归一化之后，点积和余弦相似度是等价的。
3. **欧氏距离** (Euclidean Distance)，就是中学学的两点之间的直线距离。距离越小，语义越接近。

实际场景里用得最多的是**余弦相似度**，理解起来也最直观——不用管向量长度，只管方向。

## Embedding 在 RAG 里的作用

说清楚了基础概念，回到 RAG 这个场景。Embedding 在里面扮演什么角色？

我跟你说，它就是一个翻译器。

把人类语言"翻译"成计算机能比较的数字语言。不同的是，普通翻译器是中文翻英文，Embedding 模型是自然语言翻向量。

离线的时候，把知识库里所有文档切成 chunk，每个 chunk 都用同一个 Embedding 模型转成向量，存进向量数据库。

用户提问的时候，用同一个模型把他的问题也转成向量。

然后在向量数据库里，找到和"问题向量"距离最近的 Top-K 个 chunk，这些就是召回的上下文。

最后把问题和召回的上下文一起发给大语言模型，生成答案。

整个流程就是这样，看起来不复杂，但有几个关键点你得知道。

## 决定检索质量的关键

Embedding 的质量直接决定了 RAG 检索的准确性。

一个好的 Embedding 模型，能精准捕捉问题和文档之间的语义联系。用户的问题和原文表述不完全一样？没关系，语义对了就能匹配。

一个差的模型，看不懂语义，只能做表面匹配，可能召回完全不相关的内容，反而"污染"了上下文，让大模型答非所问。

所以选好 Embedding 模型这件事，在 RAG 系统里是重中之重。

## 几个关键特性

1. **输入长度有限制**。每个模型都有最大 token 数限制，比如 512 或者 8192。超过的部分会被截断。这也是为什么之前要做分块——把长文档切成短文本块，确保每个块都不超过模型的输入上限。
2. **输出维度固定**。同一模型输出的向量维度是固定的。比如某个模型输出 1024 维，不管你输入一个词还是一段话，输出的向量长度都是 1024。
3. **同一模型内可比较**。只有用同一个模型生成的向量才能互相比较。你用模型 A 把 chunk 转成向量，检索的时候必须也用模型 A 把 query 转成向量。换模型就等于所有向量要重新生成。

第三点非常重要，我再重复一遍——换模型等于所有向量要重新生成。

## Embedding 技术发展到哪了

回顾一下 Embedding 技术的发展历程，能帮你理解现在这些模型为什么长成这样。

最早的词嵌入是静态的。Word2Vec 和 GloVe 给每个词分配一个固定的向量，不管上下文是什么。同一个词在不同句子里，向量完全一样。

好处是简单，坏处也很明显——"我吃了一个苹果"和"苹果公司发布了新手机"，两个"苹果"的语义完全不同，但向量是完全相同的。这显然不够用。

后来有了 Transformer 和 BERT，词向量变成了上下文相关的。同一个词在不同语境里生成不同的向量，一词多义的问题才真正被解决。

再后来 RAG 框架出现，对 Embedding 提出了新的要求——领域自适应、多粒度支持、混合检索。通用的嵌入模型在专业领域（比如法律、医疗）往往表现不好，需要专门微调。

## 怎么选 Embedding 模型

理论讲完了，该动手选模型了。

先看 MTEB 排行榜

[MTEB (Massive Text Embedding Benchmark)](https://huggingface.co/spaces/mteb/leaderboard)  是 Hugging Face 维护的文本嵌入模型评测基准，涵盖了分类、聚类、检索、排序等多种任务。

![Massive Text Embedding Benchmark.png](imgs/embedding/Massive%20Text%20Embedding%20Benchmark.png)

MTEB 排行榜

看这张图，选模型要权衡四个核心维度。

横轴是模型参数量 (Number of Parameters)，代表了模型的大小。通常，参数量越大的模型（越靠右），其潜在能力越强，但对计算资源的要求也越高。

纵轴是平均任务得分 (Mean Task Score)，代表了模型的综合性能。这个分数是模型在分类、聚类、检索等一系列标准 NLP 任务上的平均表现。分数越高（越靠上），说明模型的通用语义理解能力越强。

气泡大小是嵌入维度 (Embedding Size)，代表了模型输出向量的维度。气泡越大，维度越高，理论上能编码更丰富的语义细节，但同时也会占用更多的存储和计算资源。

气泡颜色是最大处理长度 (Max Tokens)，代表了模型能处理的文本长度上限。颜色越深，表示模型能处理的 Token 数量越多，对长文本的适应性越好。

MTEB 榜单能帮你快速筛选掉大量不合适的模型。但需要注意，榜单上的得分是在通用数据集上评测的，可能没法完全反映你特定业务场景下的表现。

关键评估维度

看榜单的时候，除了分数，还要关注几个维度。

任务 (Task)。对于 RAG 应用，重点看模型在 Retrieval 任务下的排名。

语言 (Language)。模型是否支持你的业务数据使用的语言？对于中文 RAG，选明确支持中文或多语言的模型。

模型大小 (Size)。模型越大性能越好，但对硬件要求也越高，推理速度也越慢。要根据你的部署环境来权衡。

维度 (Dimensions)。向量维度越高，能编码的信息越丰富，但存储和计算成本也越高。

最大 Token 数 (Max Tokens)。这决定了你分块策略的上限，块大小不能超过这个限制。

成本 (Cost)。如果用 API 服务，要考虑调用成本；如果自部署，要评估硬件消耗和运维成本。

迭代测试与优化

不要只依赖公开榜单做最终决定。

第一步，确定基线。根据上述维度，选几个符合要求的模型作为初始基准模型。

第二步，构建私有评测集。根据真实业务数据，创建一批高质量的评测样本，每个样本包含一个典型用户问题和他对应的标准答案或最相关的文档块。

第三步，迭代优化。用基线模型在私有评测集上跑，评估召回的准确率和相关性。如果效果不理想，换模型或者调整其他环节。通过几轮对比测试，选出在你的场景下表现最好的那个。

向量维度怎么选

维度是选模型的时候绑定的——你选了某个模型，维度就确定了（部分模型支持多种维度输出，但大多数是固定的）。

那维度高好还是低好？

维度就像描述一个人用了多少个特征。用 2 个特征（身高、体重）描述一个人，信息很有限，很多人会"撞衫"。用 100 个特征（身高、体重、肤色、发型、口音、走路姿势……）描述，区分度就高多了。但特征越多，记录和比较的成本也越高。

实际项目中的权衡：


| 维度范围  | 适用场景                 | 存储成本（100 万条） |
| --------- | ------------------------ | -------------------- |
| 256~512   | 简单场景，文本短、类目少 | 约 1~2 GB            |
| 768~1024  | 大多数生产场景的甜蜜点   | 约 3~4 GB            |
| 1536~4096 | 对精度要求极高的场景     | 约 6~16 GB           |

对于大多数中文 RAG 项目，768 到 1024 维是一个比较稳妥的选择。能保证足够的语义区分度，存储和检索成本也在可控范围内。除非你的场景对精度有极致要求（比如法律条文检索、医疗知识库），否则不需要上 3072 或 4096 维。

## 相似度搜索的常见算法

刚介绍过了衡量两个向量是否"相似"的算法，余弦相似度、欧氏距离这些。

但实际检索的时候，当数据量达到百万、千万甚至亿级别，逐一计算查询向量与库中所有向量的距离会变得极其缓慢。

KNN 算法是最精确的搜索方式。它会扫描所有数据，计算目标向量与数据库中每一个向量的距离，然后返回最近的 K 个。但精度是 100% 不假，计算时间也是随数据量线性增长的，不适用于海量数据。

因为 KNN 在大数据上太慢，需要更高效的索引算法来加速检索。这类算法叫做近似最近邻搜索算法 (ANN - Approximate Nearest Neighbor)，为了速度牺牲极少量的精度，是目前工业界的主流方案。

**IVF (Inverted File Index，倒排索引)** 的灵感来源于图书馆分类。如果把图书馆的书按照主题分成 N 个区域（聚类），你找书的时候先定位到最相关的几个区域，再在那个区域里挨个找，而不是满图书馆跑。

HNSW 是目前综合性能最好的算法，它利用"小世界网络"现象，构建多层的导航图。

底层（Layer 0）包含所有数据点，连接距离很近的邻居，像城市里的街道，用于精确定位。

高层（Layer 1, 2...）节点越来越少，连接距离较远的节点，像高速公路，用于快速跨越空间。

搜索的时候从顶层开始，快速定位到大致区域，然后"下楼梯"进入下一层，在下一层继续通过贪婪算法找更近的点，直到最底层。

HNSW 的优点是搜索速度极快（接近 O(logN)），召回率极高，支持动态插入数据。缺点是索引构建比较耗时，内存占用较高。

**LSH (Locality Sensitive Hashing，局部敏感哈希)**

LSH 的思路与 IVF 类似，但不依赖聚类中心，而是依赖数学上的概率哈希。

传统哈希函数是让输入差一点，输出完全不同。LSH 反其道而行，设计特殊的哈希函数，使得距离近的向量碰撞概率高。

将高维向量通过随机超平面投影等方法映射成短的二进制码或整数，然后准备多张哈希表。搜索时计算查询向量的哈希值，直接去对应的桶里把数据捞出来。因为可能漏掉，通常会查多张表或相邻的桶。

优点是理论速度最快（接近 O(1)），非常适合海量数据的粗筛。缺点是召回率不稳定，容易漏掉真正的近邻，对参数调优比较敏感。

**怎么选**

综合性能最好选 HNSW。内存非常吃紧（比如要在手机端或内存小的服务器上跑亿级数据）选 IVF_PQ。简单去重或极快速过滤可以选 LSH。

Embedding API 的请求和响应格式

在写 Java 代码之前，先用 curl 感受一下这个 API 长什么样。

请求格式

```bash
curl -X POST "https://api.siliconflow.cn/v1/embeddings" \
  -H "Authorization: Bearer 你的API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen/Qwen3-Embedding-8B",
    "input": ["小天学RAG"],
    "encoding_format": "float"
  }'
```

三个关键字段：

`model` 指定用哪个 Embedding 模型。

`input` 要转成向量的文本，可以是一个字符串，也可以是一个字符串数组（批量处理）。

`encoding_format` 向量的编码格式，`float` 表示返回浮点数。

## 实际项目里的关键决策

跑通 Demo 离生产环境还差得远。这几个问题你迟早会遇到。

云端 API 还是本地部署

云端 API 适合项目初期，数据量不大，想快速验证效果。零部署成本，按调用量付费。缺点是数据要发到第三方服务器，网络延迟也不可控。

本地部署适合数据量大、对数据安全有要求（金融、医疗、政务）、对延迟敏感的场景。数据不出内网，延迟低。缺点是要 GPU 服务器，运维成本高。

如果选本地部署，Ollama 是最简单的方案，跟云端 API 的核心逻辑一样，切换成本不高。

什么时候要重新向量化

换了 Embedding 模型。所有向量要重新生成。

文档内容更新了。对应的 chunk 向量要删掉重建。

分块策略调整了。全量重新向量化，工作量不小。

模型升级了。同一个模型的不同版本也可能不兼容。

一个实用建议：元数据里记录 `embedding_model` 和 `embedding_model_version`，这样随时知道每个向量是哪个模型生成的，升级的时候也好做灰度切换。

## 小结

这篇文章从"为什么需要向量化"出发，讲了 Embedding 原理、模型选型、相似度计算，最后跑通了一个完整的检索 Demo。

核心就几句话：

向量化是把文本转成向量，语义相近的在向量空间里距离近。

选模型看中文效果、维度、成本这三个指标。

余弦相似度最常用，值域 0-1，越接近 1 越像。

元数据不参与向量化，但要和向量一起存，检索时用来过滤。

换模型、改文档、调分块策略都要重新向量化，这些决策尽早定。

到这里，RAG 的数据准备阶段就基本完成了。文档切块、加元数据、向量化。

但向量存哪儿？几百万个向量怎么快速找到最像的那几个？
